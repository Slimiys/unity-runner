## План развития Sandbox Runner

Этот документ — **конспект по проекту**. Он помогает понять, как устроен текущий раннер, и в каком порядке мы будем его развивать.

---

### 1. Краткое описание игры

- Жанр: **3D‑раннер**.
- Игрок автоматически бежит вперёд по оси Z.
- Игрок может смещаться влево/вправо по оси X и прыгать.
- По дороге есть **монеты**, которые дают очки.
- При падении вниз с платформы игрок **респаунируется** в стартовой позиции.

Цель: собрать как можно больше монет и (в будущем) пробежать как можно дальше, не упав и не врезавшись в препятствия.

---

### 2. Текущие системы и скрипты

#### 2.1. Движение игрока

- `Assets/Scripts/Core/PlayerController.cs`
  - Отвечает за:
    - Автоматическое движение вперёд с базовой скоростью.
    - Горизонтальное управление (влево/вправо).
    - Прыжок и проверку «стоит ли игрок на земле».
    - Реализацию интерфейса `IRespawnable` (возврат в стартовую позицию).
  - Использует:
    - **Новую систему ввода** (`Input System`): метод `OnMove` получает вектор движения.
    - `Rigidbody` для физического перемещения.

#### 2.2. Камера

- `Assets/Scripts/Core/CameraFollow.cs`
  - Камера следует за игроком с фиксированным смещением и смотрит на него.
  - Логика простая и хорошо подходит для первого проекта.

#### 2.3. Монеты и очки

- `Assets/Scripts/Gameplay/Coin.cs`
  - Вращающаяся монета.
  - При попадании в триггер игрока:
    - Увеличивает счёт через сервис `IScoreService` (инъекция выполняется `CoinSpawner`).
    - Уничтожает себя.

- `Assets/Scripts/Gameplay/CoinSpawner.cs`
  - При старте сцены генерирует несколько монет:
    - В заданном диапазоне по осям X и Z.
    - С минимальным расстоянием между монетами по оси Z.
  - Показывает область спавна через гизмо в редакторе.
  - Расширяется через стратегии:
    - `ICoinLaneSelector` — выбор полосы для спавна.
    - `ICoinGroundSnapper` — привязка монеты по высоте к трассе.

- `Assets/Scripts/UI/ScoreManager.cs`
  - Синглтон, который хранит текущий счёт.
  - Реализует `IScoreService` для начисления очков без жёсткой зависимости от singleton.
  - Обновляет текстовое поле TextMesh Pro.

#### 2.4. Мир и респаун

- `Assets/Scripts/Core/IRespawnable.cs`
  - Интерфейс для объектов, которые можно «респаунить» (вернуть в исходное состояние).

- `Assets/Scripts/World/DeathZone.cs`
  - Зона‑триггер под уровнем.
  - Если в неё попал объект с тегом `Player` и реализацией `IRespawnable`, вызывает `Respawn()`.

### 3. Цели развития проекта

На основе текущего состояния мы хотим:

1. Сделать **модульную трассу** из сегментов, чтобы забег мог быть длиннее и интереснее.
2. Добавить **препятствия** (блоки, ямы, простые движущиеся объекты).
3. Заменить/дополнить визуал на **модель человека с анимацией бега**.
4. Ввести **игровые состояния** (меню → игра → конец забега) и базовую мета‑игру (UI, лучший результат).
5. Подготовить **мобильное управление** (свайпы) и адаптацию под телефоны.
6. При этом **соблюдать SOLID** и держать архитектуру понятной.

### 4. План по шагам (маршрут развития)

Каждый шаг можно делать отдельной мини‑задачей, разбирая код и логику постепенно.

#### Шаг 1. Понимание текущего кода

Цель: спокойно разобрать, **как работает прототип**, не добавляя новой сложности.

- Пройтись по:
  - `PlayerController` — как считается скорость, как работает прыжок, как проверяется земля.
  - `Coin` и `CoinSpawner` — как генерируются и собираются монеты.
  - `ScoreManager` — как устроен синглтон и обновление UI.
  - `DeathZone` и `IRespawnable` — как работает респаун.
- Нарисовать для себя небольшую схему «что за чем вызывается» во время забега.

#### Шаг 2. Сегменты трассы и генерация уровня

Цель: уйти от одной короткой дорожки к **набору повторно используемых сегментов**.

- Ввести префабы сегментов трассы в `Assets/Prefabs` (прямой участок, участок с монетами и т.п.).
- Добавить менеджер трассы (например, `TrackManager`) в `Assets/Scripts/World`:
  - Держит список префабов сегментов.
  - Подстраивает новые сегменты к концу предыдущего.
  - Удаляет старые сегменты, когда игрок далеко убежал.
- Решить, как встраивать спавн монет и препятствий в сегменты (локальные спавнеры или точки).

#### Шаг 3. Препятствия и базовая сложность

Цель: сделать так, чтобы игроку **приходилось реагировать**, а не просто бежать по прямой.

- Добавить простые типы препятствий:
  - Статические блоки, которые нужно обойти или перепрыгнуть.
  - Ямы/разрывы.
  - При желании — простые движущиеся объекты.
- Определить, что происходит при столкновении:
  - Например: респаун игрока и сброс счёта / части счёта.
- Привязать препятствия к сегментам трассы (чтобы сложность росла по мере продвижения).

#### Шаг 4. Модель человека и анимация бега

Цель: сделать визуал приятнее и **потренироваться работать с Animator**.

- Импортировать модель человека (humanoid) и набор анимаций (бег, прыжок, падение, ожидание).
- Создать префаб игрока с компонентами:
  - `Rigidbody`, `Collider`, `PlayerController`.
  - Дочерний объект с `Animator` и моделью.
- Настроить `Animator Controller`:
  - Состояния: Idle → Run → Jump → Fall.
  - Параметры: скорость, `IsGrounded`, вертикальная скорость.
- Добавить `PlayerAnimationController`, который читает данные из `PlayerController` и обновляет параметры Animator.

#### Шаг 5. Состояния игры и UI

Цель: сделать из прототипа **завершённую игру с циклами «меню → забег → конец забега»**.

- Добавить `GameManager`:
  - Состояния: `MainMenu`, `Playing`, `Paused`, `GameOver`.
  - Методы: `StartRun`, `EndRun`, `Restart`, `BackToMenu`.
- Сделать UI‑экраны:
  - Главное меню (кнопка Start).
  - HUD (очки, при желании дистанция).
  - Экран Game Over (итоговый счёт, лучший результат, кнопка Restart).
- Добавить сохранение лучшего результата через `PlayerPrefs`.

#### Шаг 6. Мобильное управление и адаптация под телефоны

Цель: подготовить игру к **комфортной игре на Android/iOS**.

- В `Input System` добавить схему управления для мобильных:
  - Свайпы влево/вправо.
  - Свайп вверх для прыжка.
- Сделать адаптер ввода, который выбирает схему в зависимости от платформы.
- Проверить, как выглядит UI на телефоне (масштаб, читаемость, кнопки).

#### Шаг 7. Рефакторинг под SOLID

Цель: научиться **держать код чистым**, когда проект растёт.

- Проверить, не делает ли `PlayerController` слишком много:
  - При необходимости вынести чтение ввода и анимацию в отдельные компоненты.
- Разделять runtime‑логику и editor‑инструменты (пример: `TrackPath` — editor‑часть вынесена в отдельный partial).
- Делать поведение расширяемым через небольшие интерфейсы (пример: `CoinSpawner` — стратегии `ICoinLaneSelector` и `ICoinGroundSnapper`, а начисление — через `IScoreService`).
- Ввести небольшие интерфейсы там, где это оправдано:
  - Для ввода (ПК vs мобильный).
  - Для сохранений (в будущем можно заменить `PlayerPrefs`).
- Сверяться с `docs/CodingStandards.md` и постепенно улучшать код.

---

### 5. Как использовать этот план

- Не обязательно делать всё подряд — можно выбирать шаги, которые сейчас интересны.
- Важно: **после каждого шага** останавливаться и осмыслять, что нового ты понял про Unity и C#.
- Если какой‑то шаг покажется слишком сложным, его можно разбить ещё на более мелкие подзадачи и описать их отдельно в этом файле.
