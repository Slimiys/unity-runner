## Правила кодирования для Unity‑проекта

### Общие принципы

- **SOLID**
  - **Single Responsibility**: каждый класс отвечает за одну чёткую задачу.
  - **Open/Closed**: поведение расширяем через абстракции, не ломая существующий код.
  - **Liskov Substitution**: наследники полностью заменяют базовый тип без изменения ожидаемого поведения.
  - **Interface Segregation**: интерфейсы маленькие и специализированные.
  - **Dependency Inversion**: высокоуровневый код зависит от абстракций (интерфейсов), а не от конкретных реализаций.

- **Компонентный подход Unity**
  - Логика разбивается на небольшие компоненты `MonoBehaviour` с одной зоной ответственности.
  - Избегать «God‑objects» и жёстких связей через `Find` / `FindObjectOfType`, по возможности использовать ссылки через `SerializeField` или абстракции.

### Организация проекта

- **Структура папок**
  - Разделять ресурсы по назначению: `Scenes`, `Scripts`, `Prefabs`, `Materials`, `Textures`, `Audio`, `Animations`, `UI`, `Editor`, `Plugins`.
  - Внутри `Scripts` группировать по областям ответственности: `Core`, `Gameplay`, `UI`, `Data`, `Infrastructure`, `Editor`.
- **Имена файлов и ассетов**
  - Скрипт и класс внутри него должны иметь одно и то же имя (`PlayerController.cs` → `PlayerController`).
  - Префабы, сцены, материалы и анимации — `PascalCase` с говорящим назначением (`MainMenuScene`, `PlayerPrefab`, `DamageTextAnimation`).
  - Для UI‑элементов использовать понятные суффиксы: `XxxPanel`, `XxxButton`, `XxxSlot`.
- **Namespace‑ы и сборки**
  - Использовать namespace, отражающий модуль/папку (`Sandbox.Gameplay`, `Sandbox.UI`), чтобы избежать конфликтов имён.
  - По мере роста проекта разбивать код на модули с помощью `.asmdef` для ускорения компиляции и лучшей изоляции.

### Именование (C#)

- **Классы, интерфейсы, перечисления**: `PascalCase`  
  Пример: `PlayerController`, `ScoreManager`, `IGameState`.

- **Публичные методы и свойства**: `PascalCase`  
  Пример: `Respawn()`, `AddScore(int amount)`.

- **Параметры методов и локальные переменные**: `camelCase`  
  Пример: `amount`, `playerRigidbody`, `groundLayer`.

- **Приватные поля**: `_camelCase`, при необходимости с `[SerializeField]`  
  Примеры: `_horizontalSpeed`, `_groundCheckRadius`, `_scoreText`.

- **Константы**: `PascalCase`  
  Пример: `PlayerTag`.

### Форматирование и стиль

- Использовать единый стиль форматирования:
  - Открывающая фигурная скобка на новой строке для классов и методов.
  - Пробелы вокруг операторов (`=`, `+`, `==`) и после запятых.
  - Не смешивать табы и пробелы; использовать единый размер отступа (обычно 4 пробела).
- Не допускать слишком длинных методов:
  - Если метод начинает выполнять несколько различных задач — разбивать на частные приватные методы.
- Избегать «магических чисел»:
  - Выносить значения в константы или сериализуемые поля с понятными именами.

### Документация

- Все **публичные классы и методы** должны иметь **XML‑документацию на русском языке**.
- Комментарии используются для объяснения намерений, ограничений и нетривиальных решений, а не для дублирования очевидного кода.

### Подход к архитектуре

- Ввод (Input), игровая логика, визуал и UI по возможности разделяются по разным компонентам/классам.
- Общие сервисы (управление состоянием игры, очками, сценами и т.п.) выносятся в отдельные менеджеры с чёткими интерфейсами.
- Зависимости между системами по возможности передаются через конструктор, свойства или `SerializeField`, а не ищутся глобально.
- Регулярно проводить небольшие рефакторинги: после добавления функциональности приводить код к стандартам, устранять дублирование, улучшать читаемость без изменения поведения.
